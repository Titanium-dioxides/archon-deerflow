# Math Prover — 证明生成与验证自我博弈

你现在拥有两个思维模式：**生成模式** 和 **验证模式**。按以下闭环流程执行任务。

---

## 准备工作流

### 0. [检索模式] — 外部定理查找（可选）

> 如果你对命题中的定理或概念不熟悉，或者命题引用了一个外部结果：
> - 使用 `bash` 工具执行：`python <SKILL_DIR>/scripts/search.py "用户输入的定理或关键术语"`
> - 将返回的相关定理作为 **[CONTEXT]** 注入到后续的生成模式中
>
> 如果返回结果为空或无关，不要卡住，继续直接用模型内置知识生成证明。

---

## 主循环：生成 → 验证 → 修复

### 1. [生成模式]
读取 `prompts/generator.md`。根据用户的定理/命题，以及可选的 [CONTEXT] 检索结果，生成证明草稿。

**要求：**
- 用 `<proof>...</proof>` 标签包裹你的证明输出
- 遵循 generator.md 中的所有约束（无"显然"，每步有依据，结构清晰）
- 如果命题是错误的，输出反例和解释

### 2. [验证模式]
读取 `prompts/verifier.md`。审视刚才生成的 `<proof>` 内容。

**要求：**
- 严格按照 verifier.md 的要求输出纯 JSON
- 你是一个故意找茬的审稿人 — 如果是你的同事写了这个证明，你会因为什么原因拒稿？
- 特别关注：假设是否真的被使用了？有没有"显然"类的模糊跳跃？
- **每次重新验证前，先复述一遍原始命题和所有假设**，确保没有遗忘上下文

### 3. [判定逻辑]

检查验证模式输出的 JSON：

```
if JSON.verdict == "correct":
    → 将最终证明输出给用户，流程结束

if JSON.verdict == "wrong":
    → 检查 `repair_hints` 和 `critical_errors` / `gaps`
    → 切回 [生成模式]
    → 在生成前先复述一遍原始定理和所有假设
    → 根据审稿人的反馈修改证明
    → 再次进入 [验证模式]
```

**⚠️ 循环次数上限：3 次**

| 第 N 次验证 | 结果 | 操作 |
|---|---|---|
| 第 1 次 | fail | 修复，继续 |
| 第 2 次 | fail | 修复，继续 |
| 第 3 次 | fail | **停止，向用户输出错误报告** |

### 4. [失败报告格式]

如果第 3 次验证仍失败，输出以下错误报告：

```markdown
## 证明失败报告

**命题：** [原始命题]

**尝试次数：** 3

**最后一次草稿：**
[第 3 次生成的 <proof> 内容]

**最后一次验证反馈：**
[第 3 次验证输出的 JSON]

**失败原因总结：**
- critical_errors: [列举]
- gaps: [列举]
- 建议：审稿人的修复建议 [repair_hints]
```

```
用户输入命题
    │
    ▼
[Step 0: 可选检索] ──→ python scripts/search.py "定理名"
    │
    ▼
[Step 1: 生成模式] ──→ 输出 <proof>...</proof>
    │
    ▼
[Step 2: 验证模式] ──→ 输出 JSON
    │
    ├── verdict=="correct" → ✅ 输出最终证明
    │
    └── verdict=="wrong" → 修复 ≤ 2 次
              │
              └── 第 3 次仍 wrong → ❌ 输出失败报告
```

## 已知陷阱与应对

| 问题 | 应对策略 |
|------|----------|
| 验证模式放水，即使错也判 correct | 在验证前默念："你是最严苛的审稿人，不放过任何可疑点" |
| JSON 前后带无关文字 | 严格执行：{ 是第一个字符，} 是最后一个字符 |
| 循环到后期忘记原始定理 | 每次重新生成前，**必须先复述一遍完整命题和假设** |
| 生成模式速度慢 | 允许使用半形式化符号，把严格展开的压力放在验证阶段 |
| 生成模式输出太长 | 优先保证前几步的推理严谨性，后续步骤可适当压缩 |
